## Architecture decisions

**Agent Bus (A1)** — Why a new module instead of extending workflows.py?
- workflows.py is a DAG executor (one-shot per run). What we need is a long-lived
  pub/sub channel where heterogeneous agents (workflow nodes, Slack bot, Telegram
  bot, TUI live view, frontend SSE) publish and subscribe independently.
- Storage: single SQLite table on the existing DB_PATH (no new file). Append-only,
  bounded by retention (default 7 days, env-tunable).
- Hot path: in-memory deque with `threading.Condition` — subscribers `wait_for`
  new events, no polling. Cold path: `since_id` query against SQLite for
  reconnects.
- Dedup: `LRU{(topic, sha1(payload)): ts}` of size 64. Drops only exact repeats
  inside a 5s window — preserves legitimate repeated status events further apart.
- Optimization rationale: avoid per-tick polling (was the obvious naive design),
  collapse repeated "still working..." spam, and cap memory.

**Telegram (A2)** — long-poll vs webhook?
- Long-poll (getUpdates with timeout=25) is the lighter default: no public URL,
  no TLS, no port forwarding. One background thread per configured bot.
- Webhook path is opt-in via config flag (mode: "longpoll" | "webhook"); the
  webhook handler is wired in routes.py as `/api/telegram/webhook/<secret>`.

**Orchestrator (A3)** — planner / dispatcher split.
- Receives a message (channel, user, text). Resolves a "binding" (channel →
  workflow_id | ad_hoc).
- Ad-hoc: Claude (default planner model, configurable) is asked to produce a
  JSON plan: list of {assignee, prompt} steps. Sub-agents run in parallel via
  ThreadPoolExecutor (size = min(plan_len, cpu*2)).
- Each sub-agent publishes start/progress/done events to agent_bus on topic
  `orch.<run_id>`. The orchestrator subscribes, posts batched updates back to
  the channel (debounced at 2s) to avoid Slack rate limits.
- Final aggregation: another small Claude call summarises sub-agent results.

**TUI (A4)** — curses, stdlib only.
- One screen, three columns: section list / item list / detail+edit.
- Reuses the same JSON files the web UI writes to — no shadow store.

**Hardcoding ban** — concrete rules followed:
- Models: `orchestrator.json` carries `plannerModel` and `defaultAssignees`;
  no string like "claude-opus-..." in code.
- Channels: bindings stored as data, never literal in code.
- Limits: pool size, debounce ms, retention days all env-overridable in
  `server/config.py`.
- API hosts: only `slack.com` and `api.telegram.org` whitelisted; same SSL
  context pattern as slack_api.py.
